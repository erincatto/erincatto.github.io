Contact Recycling

Contact Recycling is a technique I’ve developed for Box2D and Box3D. First you might be wondering what is Box3D? More on that later.

Contact Recycling can drastically improve rigid body simulation performance. It can make the process of computing contact points almost free. Further it can significantly improved stability.

Contact Recycling is inspired as a combination of other techniques. I have not seen this specific technique used in other physics engines. There are similar solutions to the same problem but they have some fundamental differences. I’ll try to make the differences clear. It is possible this specific technique already exists in some engines. Let me know and I’ll update this blog post.

Motivation

The motivation for contact recycling is straight forward. Often when there are many rigid bodies stacked together, there may be parts of the stack that are not moving much. Yet other parts of the stack are moving a lot and keeping the whole island awake. These stationary bodies should not have their contact points recomputed every timestep. Can we devise a way to re-use the contact points from the previous time step? If so there can be massive performance gains because computing contact points is expensive. We should try to recycle them!

We can take this further. Consider a box resting on a train. Both are moving, but not relative to each other. These contact points should be recycled as well.

Before I explain contact recycling there is some background knowledge and terminology that is important to review.

Contact manifolds

A contact manifold is the surface of intersection between colliding convex shapes. A manifold for 2D collisions it is a point or a line segment. A manifold for 3D collisions is a point, line segment, or a convex 2D polygon. Usually a line segment manifold will be represented by contact points at the end. For polygon manifold in 3D the contact points are at the vertices. In all cases the contact points all have the same normal vector, the manifold normal.

One-shot contact manifold

A one-shot contact manifold algorithm computes all the contact points for a manifold under one function call within one time step. Historically this implied using the Separating Axis Test (SAT) and Sutherland-Hodgman clipping. However, people have worked out how to this with GJK by identifying convex hull features from the GJK simplex. GJK requires a fallback when there is overlap, such as SAT or the Expanding Polytope Algorithm (EPA).

Persistent contact manifold

Persistent contact manifold (PCM) algorithms seek to spread the work of building a contact manifold over several time steps. Early versions proceeded as follows:

  1. Compute closest point using GJK with EPA fallback
  2. Add point from 1 to manifold
  3. Look for a point to remove from the manifold using some heuristic

Step 3 is necessary to keep the number of contact points small. Often contact points are stored in local coordinates so that they move with the simulation. So boxes sliding across each other retain contact points.

Avoiding computation

One-shot manifolds are clearly expensive. Even PCM requires a call to GJK every time step. GJK can be slow if the shapes are complex. By the way, I consider a 2D hull with over 8 points or a 3D hull with over 32 points to be complex.

The narrow phase can detect the shapes or bodies haven’t moved much relative to each other and simply skip the manifold update and re-use the result from the previous step. Then it becomes a problem of finding a good heuristic for significant movement. For this, I’ll invoke an old friend.

Conservative Advancement

Conservative Advancement (CA) is traditionally a technique for continuous collision detection (CCD). The idea it to create an upper bound on the movement of two shapes relative to each other. Using this bound and the distance between the shapes, the CCD solver can iteratively move the bodies forward in time until they just touch or completely miss.

The premise of CA is that you can create a bound on the relative movement of two bodies (A and B) using:

$$(v_B - v_A) \cdot n + ||\omega_A|| r_A + ||\omega_B|| r_B$$

Here \(n\) is the normal vector between the closes points. The parameter \(r\) is the radius of the bounding circle centered on the center of mass (the rotation pivot).

This expression can be generally useful because it gives a way to combine linear and angular movement into a dimensionally consistent value. For example, Box2D uses this expression to measure the movement speed of a body for sleep and for CCD culling:

float maxVelocity = b2Length( linearVelocity ) + b2AbsFloat( angularVelocity ) * body->maxExtent;

This gives the maximum velocity of any point on the body. I can compare this with the sleep threshold to determine if the body is sleepy. This is great because I don’t need a separate sleep threshold for angular velocity. This makes it so long skinny bodies don’t fall asleep too soon.

I can also compare maxVelocity to the inner circle of the smallest shape on the body to determine if the body is moving fast.

bool isFast = maxVelocity > safetyFactor * body->minExtent;

Here minExtent is that inner radius and safetyFactor is some value between 0 and 1. If isFast == false then CCD can be skipped for this body. So one measure of speed can be used for two tests: sleepiness and speediness.

Relative movement

The CA formula can be used for contact pairs to determine if the relative movement is significant. This gets conceptually tricky but the code is simple. Suppose we just computed a one-shot contact manifold. At this configuration we have the transform of the two bodies:

b2Transform cachedTransformA = bodyA->transform;
b2Transform cachedTransformB = bodyB->transform;

The relative transform is computed using the math library:

b2Transform cachedRelativeTransform = b2InvMulTransforms(cachedTransformA, cachedTransformB);

The order doesn’t matter, it just needs to be consistent. When there is a fresh contact I cache this relative transform and set a flag to indicate the transform has been cached. Then the next time step I compute the current relative transform.

b2Transform relativeTransform = b3InvMulTransforms(cachedTransformA, cachedTransformB);

Abbreviate the cached relative transform as xfc and the current transform as xf. With these I can compute the relative movement of the bodies across the time step:

float relativeMovement = b2Distance(xfc.p, xf.p) + AngularDistance(xfc.q, xf.q) * maxExtent;

The AngularDistance function depends on how the rotations are represented. I take some shortcuts on this to avoid trigonometry. In Box2D rotations are represented as cosine and sine pairs. I just use the absolute value of the sine term, which makes use of the small angle approximation:

$$sin(\theta) \approx \theta$$

I do something similar with quaternions in 3D.

With all of that, I now have a single number, the relativeMovement, that indicates the frame over frame movement of the bodies relative to each other. If this is small enough then I can recycle the contacts.

A word of caution: do not update the cached relative transform when recycling. This causes drift. Only cache the relative transform on the step where the fresh manifold is computed.

Problem solved? Not quite. If you implement this you will find that you need a tight tolerance on the relative movement. If you loosen the tolerance then the simulation degrades quickly into a jitter fest. Do we give up? No, we evolve.

Updating separation

Box2D uses a solver I dub SoftStep. The idea is to use sub-steps instead of iterations and to use soft constraints to manage position error. So it uses soft sub-steps, or SoftStep for short.

A key ingredient of the sub-stepping is to avoid recomputing contact points. Otherwise I might as well just tell the user to use one iteration and small time steps. This works, but it is slow. So across sub-steps, the contact points are re-used. For each sub-step I use the relative movement of the bodies and adjust the contact point separation value, which is negative when shapes overlap. This ensures shapes are pushed apart correctly. This is much cheaper than re-computing the contact points and leads to excellent stacking stability. Better than what can be achieved with a traditional iterative solver.

I borrowed this idea of contact recycling across sub-steps by adjusting the separation values, and applied this to contact recycling across full time steps. When the contact points are computed fresh then I cached the initial separation values (baseSeparation). As the contact points are recycled, I adjust the current separation value by using the body transforms and the baseSeparation. This code is compact:

if ( distance + maxExtent * b2AbsFloat( qr.s ) < tolerance )
{
    b2Rot dqA = b2MulRot( transformA.q, b2InvertRot( cachedTransformA.q ) );
    b2Rot dqB = b2MulRot( transformB.q, b2InvertRot( cachedTransformB.q ) );
    b2Vec2 normal = manifold.normal;
    b2Vec2 dc = b2Sub( bodyB->centerB, bodyA->centerA );

    for ( int i = 0; i < manifold.pointCount; ++i )
    {
        b2ManifoldPoint* mp = manifold.points + i;
        b2Vec2 rA = b2RotateVector( dqA, mp->anchorA );
        b2Vec2 rB = b2RotateVector( dqB, mp->anchorB );
        b2Vec2 dp = b2Add( dc, b2Sub( rB, rA ) );
        mp->separation = mp->baseSeparation + b2Dot( dp, normal );
    }
}

For the math to work out, I need to cache the transforms of both bodies, not just the relative transform. But the effect is glorious. Suddenly I can use very large tolerances, up to several centimeters. I observe increased stacking stability by recycling contact points this way. And it is thanks to updating the contact point separation values.

As with many things in game physics, the best way to understand the value is to see the behavior and performance in practice. So watch this video where I show how contact recycling works in practice and some of the artifacts it has when pushed too far.

Summary

Contact recycling has been used for sub-stepping by multiple physics engines, including Bepu and PhysX. However, as far as I know, it has not been used across time steps. This can be bolted on to any physics engine that can produce one-shot manifolds. The physics engine will easily achieve higher performance and stability with low effort. Having a sub-stepping solver is not required. I’m not sure contact recycling would work will with PCM. Maybe someone will try it.
